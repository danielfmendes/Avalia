% ---------- 1. Introduction ----------
\section{Introduction}\label{sec:intro}

European housing markets have decoupled: Lisbon, Berlin, and Amsterdam outpace their national averages. Portugal's INE house-price index is up roughly 50\,\% since 2019~\parencite{INE}, yet no public tool reveals which parish drove it.

Idealista, Imovirtual, and Casa Sapo only show \emph{current} listings. None surfaces parish-level (\emph{freguesia}) price history or the signals that move the market.

Avalia makes these dynamics transparent at the level at which people transact -- the parish. Cloudflare D1's free tier caps writes at \textbf{100\,000 rows per day}~\parencite{cloudflareD1}, so we scope the prototype to the eighteen Lisbon AML municipalities, where aggregation compresses $\sim$3\,M raw rows to $\sim$65\,k. The same pipeline extends to other European cities once Lisbon is validated.

\textbf{Define the project scope through User Stories.}
\begin{itemize}
  \item As a \emph{renter or buyer}, I want parish price history so that I can negotiate from evidence.
  \item As a \emph{landlord or agent}, I want year-on-year parish benchmarks so that I can price competitively.
  \item As a \emph{municipal planner}, I want housing-pressure signals so that I can target interventions.
\end{itemize}

% ---------- 2. System ----------
\section{System architecture and reproducibility}\label{sec:system}

A renter, an investor, and a municipal planner walk into the same dataset with different questions, so Avalia is split into seven purpose-built pages rather than one catch-all view:
\begin{itemize}
  \item \emph{Market Overview}: district prices, KPIs, drill-down map.
  \item \emph{AI Predictions}: 12-month per-region forecast with uncertainty bands.
  \item \emph{Signals}: price overlaid on Google Trends, dormidas, earnings, construction.
  \item \emph{Compare}: side-by-side, up to six regions.
  \item \emph{Rooms}: T0--T4+ trends per typology.
  \item \emph{Affordability}: budget slider ranking parishes in reach.
  \item \emph{Time Machine}: monthly scrubber back to 2012.
\end{itemize}

\section{Data sources}\label{sec:data}

The MVP is built on the three Lisbon-scoped feeds listed below. Additional national portals and historical web archives are folded in at Step~2 of the rollout (Section~\ref{sec:growth}) as Avalia extends to other European cities.

\begin{itemize}
  \item \textbf{arquivo.pt + habitacao.net.} \emph{arquivo.pt} (Portugal's web archive) snapshots Idealista, Imovirtual, and Casa Sapo periodically, and \emph{habitacao.net} mines those snapshots to reconstruct listings history back to 2012 with price, area, rooms, and location per entry.
  \item \textbf{Google Trends.} Weekly Lisbon housing-keyword search interest, a leading indicator: spikes in queries like ``comprar casa'' precede price moves by one to two months.
  \item \textbf{PORDATA.} Per-municipality series: tourism overnights (dormidas, monthly) as a short-term-rental-pressure proxy, plus four annual macro series (foreign workers, tourist capacity, mean earnings, completed constructions).
\end{itemize}

\vspace{-2.6em}
\section{Forecasting model}\label{sec:forecast}\label{sec:differentiation}

Per region, the frontend fits OLS to the trailing 18 monthly €/m\textsuperscript{2} averages and projects 12 months ahead client-side, with the uncertainty band widening 1.5\,\% per step. The \emph{AI Predictions} page reruns the model per municipality and surfaces a ranked table -- sparkline, projected price, expected $\Delta$\%, vs.-district flag, confidence tied to history depth -- so a user drills from the district forecast to which municipalities pull it up or down.

\textbf{Two derived metrics ship alongside.} \emph{Rental yield} $=$ $(\text{monthly rent} \times 12) / \text{sale price}$ on 2023 weighted means, exposed on \emph{Affordability} and \emph{Compare}. \emph{YoY change} is benchmarked only against the calendar months observed in the current period, avoiding spurious dips from incomplete data.

\textbf{Where exogenous signals fit today.} The model extrapolates each region's own past prices and misses turning points. Google Trends, dormidas, and the PORDATA macro series are rendered on the \emph{Signals} page as overlays for visual correlation against price. Folding them into the forecast itself is left for future work.

\clearpage
\section{Growth estimate}\label{sec:growth}

Two rollout steps stage the geographic expansion. Each step adds a one-shot historical backfill at rollout, then keeps growing from a steady weekly scrape:
\begin{itemize}
  \item \textbf{Step 1} (national rollout: Porto, Algarve, rest of Portugal, 2027): lifts the cumulative footprint from 2.6 to 17\,GB stored and from 6 to 31\,TB processed.
  \item \textbf{Step 2} (ten major European cities, 2029): lifts it to 58\,GB stored and 100\,TB processed. Each new country adds its own historical web archive (UK Web Archive, Internet Archive Wayback, BNE), its own forward portal (Rightmove, Immoscout24, SeLoger, Habitaclia, Idealista.it), and an Airbnb scrape per market as a short-term-rental signal complementing the long-term sale and rental listings.
\end{itemize}

\textbf{Per-listing footprint} (measured on the existing Lisbon dataset):
\begin{align*}
\sigma_{\text{stored}}    &= \tfrac{2.6\,\text{GB}}{5.7\,\text{M listings}} \approx 460\,\text{B per listing kept in the aggregated tables}, \\
\sigma_{\text{processed}} &\approx 1.5\,\text{MB per listing fetched (HTML + assets) on each weekly snapshot~\parencite{httparchive2024}.}
\end{align*}

\textbf{End-2030 totals.} The end-2030 footprint of each region $r$ has two parts: a one-off backfill at rollout, plus a steady monthly increment from the forward scrape. We therefore write
\[
S_r \;=\; \underbrace{B_r}_{\substack{\text{backfill}\\\text{stored (GB)}}}
       \;+\; \underbrace{f_r}_{\substack{\text{monthly forward}\\\text{rate (GB/month)}}}\!\cdot\, m_r,
\qquad
P_r \;=\; \underbrace{C_r}_{\substack{\text{backfill}\\\text{processed (TB)}}}
       \;+\; \underbrace{g_r}_{\substack{\text{monthly forward}\\\text{rate (TB/month)}}}\!\cdot\, m_r,
\]
where $m_r$ is the number of months the region has been live by end-2030: Lisbon since June 2026 ($m=55$), the national step since July 2027 ($m=42$), the EU step since July 2029 ($m=18$). Summing the three regions:
\begin{align*}
S_{\text{end-2030}} &= \underbrace{2.6 + 0.10 \cdot 55}_{\text{Lisbon: 8.1}}
                   + \underbrace{5.6 + 0.22 \cdot 42}_{\text{National: 14.8}}
                   + \underbrace{26.0 + 0.50 \cdot 18}_{\text{EU: 35.0}}
                   \approx 58\,\text{GB}, \\[2pt]
P_{\text{end-2030}} &= \underbrace{6.0 + 0.18 \cdot 55}_{\text{Lisbon: 15.9}}
                   + \underbrace{9.5 + 0.38 \cdot 42}_{\text{National: 25.5}}
                   + \underbrace{50.0 + 0.50 \cdot 18}_{\text{EU: 59.0}}
                   \approx 100\,\text{TB},
\end{align*}
giving overall compression $P/S = 100\,\text{TB} / 58\,\text{GB} \approx 1\,730\times$. The Figure~\ref{fig:growth} gap between line and stack-top from mid-2029 is the European-cities backfill (1.92$\times$ processed/stored, denser than the forward-scrape).

% ---------- 6. Five Vs ----------
\section{The 5\,Vs of Big Data: today and at scale}\label{sec:fivevs}

\textbf{Why this is a Big Data project.} Avalia today extracts 2.6\,GB of curated Portuguese listing metadata from roughly 6\,TB of \emph{arquivo.pt} page snapshots. Through two rollout steps the production system processes \textbf{100\,TB of raw archive and weekly portal pages} to extract \textbf{58\,GB of structured insight} -- a 1\,730$\times$ compression. \emph{Volume} comes from the touched-data side. \emph{Variety} hardens in Step 2, where each new country brings its own historical web archive and forward portal. \emph{Velocity} is the weekly snapshot cadence, \emph{Veracity} is cross-portal reconciliation, and \emph{Value} is forecast + signals. The five Vs are detailed in the table below.

\begin{table}[H]
\centering

\textbf{What this needs at scale.} The Cloudflare D1 prototype does not survive Step 2. A production build swaps in S3 with Apache Iceberg for the 100\,TB archive, Apache Spark or Dask for parallel ETL across snapshots, Kafka for the weekly Idealista feed, and Airflow for orchestration. The serving layer keeps its current shape -- only the data plane changes.

% ---------- 7. Discussion ----------
\section{Discussion and conclusion}\label{sec:roadmap}

Three revenue lines, each backed by a feature already shipped:
\begin{itemize}
  \item Freemium consumer tier: \emph{Time Machine} and \emph{Affordability} free, with a paid \emph{investor} mode that unlocks the per-municipality forecast table and rental-yield calculator.
  \item B2B API for banks and real-estate agencies, seeded by the public \texttt{/api/search} endpoint (geography $\times$ typology $\times$ area filters, weighted-mean aggregates).
  \item Anonymised market reports for municipalities, driven by the \emph{Signals} page overlays (tourism, search interest, earnings, construction).
\end{itemize}
Next business steps: a paid validation pilot with one B2B partner (a small agency or appraisal firm) to anchor API willingness-to-pay, tiered usage-based pricing, and a GDPR / data-licensing review before any municipal contract.

\subsection{Data pipeline (Python ETL)}

The pipeline is an offline, run-on-demand step that converts a single curated CSV into the SQL inserts that are then pushed to D1.

\prompt{Write a small Python script that reads csv/summary\_lisboa\_final.csv and emits batched INSERT INTO habitacao (...) statements into ../backend/sql/Insert\_Queries.sql. The CSV columns map 1:1 to the table columns: mes\_ano, tipo\_venda, tipo\_habitacao, quartos, distrito, municipio, freguesia, total\_rows, avg\_area, avg\_preco, avg\_m2. Group rows into batches of 100 to avoid SQLITE\_TOOBIG when loading into Cloudflare D1. Escape single quotes by doubling them, leave numeric values unquoted, wrap text in single quotes.}

\subsection{Database schema (Cloudflare D1)}

A single pre-aggregated table is enough, because aggregation has already been done in the ETL step.

\prompt{Design a Cloudflare D1 schema for aggregated Portuguese housing listings. One table called habitacao, columns: mes\_ano (TEXT, "YYYY-MM"), tipo\_venda, tipo\_habitacao, quartos (TEXT, numeric string like "0.0"), distrito, municipio, freguesia, total\_rows (INTEGER, used as a weight when re-aggregating), avg\_area, avg\_preco, avg\_m2 (REAL). \subsection{Problem encountered: Lisboa-level aggregation}

Early in the build the district-level view showed zero rows for Lisbon city. Inspecting the data, Lisboa turned out to be the only municipality whose pre-aggregated ``Grouped at Municipio level'' marker rows are absent from the source dataset. Every other municipality has them, but for Lisboa only the underlying parish rows exist. A naive \texttt{WHERE freguesia = 'Grouped at Municipio level'} therefore filters Lisboa out entirely, leaving a blank tile on the district map. The fix synthesises Lisboa's municipality row on the fly at query time, weighted by \texttt{total\_rows}. Cloudflare D1, which is SQLite-based and runs at the edge, does not handle correlated subqueries reliably for this shape, so the implementation uses \texttt{UNION ALL}. The resulting branch lives in \texttt{api/src/index.ts} under the \texttt{level=district} path.

\prompt{In our D1-backed Hono Worker, GET /api/search?level=district returns no rows for Lisbon city. \subsection{Secondary problem: parish-name mismatch on the Lisbon map}
